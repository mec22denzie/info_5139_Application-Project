## CRITICAL

### 1. Exposed Firebase Credentials in .env
**File:** `.env` (committed to git history)
**Issue:** All Firebase config values (API key, project ID, database URL, app ID) are in `.env`. While `.gitignore` includes `.env`, the file was already committed to the repo and exists in git history.
**Impact:** Anyone with repo access can extract credentials and impersonate the app against your Firebase backend.
**Fix:**
1. Rotate ALL Firebase API keys in Firebase Console immediately
2. Remove `.env` from git history: `git filter-branch --force --index-filter "git rm --cached --ignore-unmatch .env" -- --all`
3. Consider enabling Firebase App Check to restrict API usage to legitimate app builds

---

### 2. No Firestore Security Rules
**File:** No `firestore.rules` file found in project
**Issue:** Without server-side Firestore rules, ALL collections are likely open to any authenticated (or unauthenticated) user. Client-side filtering provides zero security.
**Impact:** Any user can read/write any document in any collection — orders, users, admin actions, notifications — by making direct Firestore SDK calls.
**Fix:** Deploy Firestore security rules. Minimum viable rules:
```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId} {
      allow read: if request.auth.uid == userId;
      allow write: if false; // only Cloud Functions can modify
    }
    match /orders/{orderId} {
      allow read, write: if request.auth.uid == resource.data.userId;
    }
    match /carts/{cartId} {
      allow read, write: if request.auth.uid == resource.data.userId;
    }
    match /products/{productId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if request.auth.uid == resource.data.donorId;
    }
    match /notifications/{notifId} {
      allow read: if request.auth.uid == resource.data.userId;
      allow create: if request.auth != null;
      allow update: if request.auth.uid == resource.data.userId;
    }
    match /reports/{reportId} {
      allow create: if request.auth != null;
      allow read, update: if false; // admin-only via Custom Claims
    }
    match /systemLogs/{logId} {
      allow create: if request.auth != null;
      allow read: if false; // admin-only via Custom Claims
    }
    match /adminActions/{actionId} {
      allow read, write: if false; // admin-only via Custom Claims
    }
  }
}
```

### 3. Client-Side Role Enforcement Only — No Server Validation
**Files:** `App.js:50-66`, `adminService.js` (all functions), `moderationService.js` (all functions)
**Issue:** Role-based access is enforced only by client-side React Navigation routing. Admin service functions (`disableUser`, `enableUser`, `changeUserRole`, `removeListing`, `dismissReport`) accept an `adminUid` parameter but never verify the caller is actually an admin.
**Impact:** Any authenticated user can call `changeUserRole(myUid, 'Admin', myUid)` to promote themselves, then access all admin functions.
**Fix:**
1. Use **Firebase Custom Claims** to set admin role at the token level (requires Cloud Functions)
2. Add Firestore rules that check `request.auth.token.admin == true` for admin-only operations
3. As a short-term mitigation, add client-side verification in each admin function:
```javascript
const callerDoc = await getDoc(doc(firestore, 'users', auth.currentUser.uid));
if (!callerDoc.exists() || callerDoc.data().role !== 'Admin') {
  throw new Error('Unauthorized: Admin role required');
}
```

## HIGH

### 4. Unauthorized User Data Exposure
**Files:** `adminService.js:20-28` (`getAllUsers`), `adminService.js:32-41` (`getUserById`)
**Issue:** `getAllUsers()` and `getUserById()` have no authorization check. Any code path that calls these functions exposes all user records (email, name, role, status).
**Fix:** Add admin role verification before returning data. Enforce with Firestore rules.

---

### 5. Cart/Order Price Manipulation
**Files:** `ProductDetailScreen.js:28-34`, `CheckoutScreen.js:111-120`
**Issue:** Product prices are stored in the cart from client-side state. The checkout total is calculated client-side from cart data. A user could manipulate prices in React state before checkout.
**Impact:** Orders placed with incorrect totals.
**Fix:** On checkout, re-fetch each product's price from Firestore and recalculate server-side. Ideally use a Cloud Function for order creation.

---

### 6. No Ownership Check on Product Edit/Delete
**Files:** `MyListingsScreen.js` (delete function), `EditItemScreen.js` (update function)
**Issue:** When editing or deleting a product, the code does not verify `product.donorId === auth.currentUser.uid`.
**Fix:** Add ownership check before `updateDoc`/`deleteDoc`. Enforce with Firestore rules: `allow update, delete: if resource.data.donorId == request.auth.uid`.

---

### 7. Stack Traces in Persistent Logs
**Files:** `errorLogger.js:40-44`
**Issue:** Full error stack traces are written to the `systemLogs` Firestore collection. These reveal internal file paths, library versions, and application logic.
**Fix:** Truncate or omit stack traces in production. Only store the error message and code:
```javascript
stackTrace: __DEV__ ? error.stack : null,
```

---

### 8. No Auth Check in Admin/Moderation Service Entry Points
**Files:** `adminService.js` (all exports), `moderationService.js` (`dismissReport`, `removeListing`)
**Issue:** None of the exported functions check `auth.currentUser` before proceeding. If called without authentication context, they silently execute with `userId: null`.
**Fix:** Add `if (!auth.currentUser) throw new Error('Not authenticated');` at the top of every sensitive function.

---

## MEDIUM

### 9. Notification Content Leaks Email Addresses
**Files:** `PostItemScreen.js:134-140`, `adminService.js` (notification calls)
**Issue:** Notification messages include user email addresses in plain text (e.g., `"Item posted by user@email.com"`). These are stored permanently in the `notifications` collection.
**Fix:** Use generic messages instead: `"A new listing was posted — check the moderation queue"`.

---

### 10. No Rate Limiting on Reports, Posts, or Notifications
**Files:** `ReportListingScreen.js`, `PostItemScreen.js`, `notificationService.js`
**Issue:** Duplicate report prevention exists per-user-per-product, but no global rate limit. A user could file reports against hundreds of products, spam listings, or trigger notification floods.
**Fix:** Add per-user daily limits. Track counts in Realtime DB or use Cloud Functions with rate limiting.

---

### 11. Password Policy is Client-Side Only
**Files:** `SignUpScreen.js:48-50`, `src/utils/validation.js`
**Issue:** Password strength (`isStrongPassword`) is only checked on the client. Direct Firebase Auth API calls bypass it.
**Fix:** Configure Firebase Auth password policy in Firebase Console (Settings > Authentication > Password policy). Firebase does enforce a 6-char minimum server-side by default.

---

### 12. Console Logging of Sensitive Data in ProductListScreen
**Files:** `ProductListScreen.js` (multiple `console.log` calls)
**Issue:** Extensive debug logging remains: Firestore status, product names, snapshot sizes. This leaks app internals to anyone with DevTools.
**Fix:** Wrap in `__DEV__` checks or remove entirely:
```javascript
if (__DEV__) console.log("Debug info");
```

---

## LOW

### 13. Missing Input Length Limits on Checkout Fields
**Files:** `CheckoutScreen.js:82-90`
**Issue:** First name, last name, email, and phone have format validation but no max length. Extremely long strings could cause UI overflow or excessive Firestore storage.
**Fix:** Add `maxLength` prop to `TextInput` components and validate length before submit.

---

### 14. Product Filter Logic Could Expose Removed Products
**Files:** `ProductListScreen.js:249-273`
**Issue:** Products with `status === "removed"` are filtered client-side. The Firestore query fetches ALL products first, then filters. Removed product data still transits the network.
**Fix:** Add `.where("status", "!=", "removed")` to the Firestore query itself, or better: `.where("status", "==", "active")`.
